Best Practices for Secrets Handling
Never commit Secret YAML with real values to Git. The base64 in the manifest is trivially reversible. Anyone with repo access has the secret. Use placeholder values for learning manifests only.
Use stringData instead of data in manifests. It accepts plain text and Kubernetes encodes it for you — less error-prone than encoding manually:
stringData:
  DB_PASSWORD: supersecret123   # plain text, Kubernetes handles encoding
Prefer volume mounts over env vars for sensitive values. Env vars can leak in crash logs, debug output, and child process environments. Files in a mounted volume are easier to control and can be updated without restarting the Pod (Kubernetes updates the file automatically within ~1 minute).
Enable encryption at rest. By default etcd stores Secrets in plain base64. In production, enable EncryptionConfiguration in your cluster to encrypt etcd data using AES or a KMS provider.
Use RBAC to restrict access. Only the service accounts that genuinely need a Secret should be able to get or list it:
# Check who can read your secret
kubectl auth can-i get secret/my-app-secret \
  --as=system:serviceaccount:my-app-namespace:default
For production, use an external secrets manager. Kubernetes Secrets are convenient but limited. Tools like External Secrets Operator (ESO) pull secrets from AWS Secrets Manager, HashiCorp Vault, or GCP Secret Manager into the cluster automatically, giving you rotation, auditing, and centralized control.
